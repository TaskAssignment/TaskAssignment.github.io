Welcome to TaskAssignment

This is the needed documentation to continue the development of this software. It's intended to help developers with previous information and assumptions made throughout the development course.

One important architectural decision is the profile division for Developers and Bugs. They are similar and yet different. These abstractions are needed to analyze data from different sources, like GitHub, Bugzilla, StackOverflow. More details on each one bellow.

Bugs

Bugs can come from GitHub or Bugzilla, as of now. Those are considered subprofiles and reference a Bug that will contain the basic information for a bug analized in this system. Check the Bug documentation to know what is this basic information.

For now, a Bug must be created just before the creation of a specific profile. The specific profile must, then, reference the general Bug . Because this is not an issue tracker, and we fetch bugs from other places, Bug _id must be the a 2 letter acronym representing the origin plus the specific id. Example: A Bug from GitHub has an id equals to 123. If I fetch this bug to the system database, I'll have a GitHubIssue with _id 123 that will reference a Bug with _id 'GH123'. A Bug from Bugzilla with the same id, would reference a Bug with _id 'BZ123'.

This ensures that bugs from different sources with the same id will have different ids on our database. It's also an easy way to check the source of a bug. And, because an id is already unique inside a system, like a GitHub id, we can safely refer to them when any other information on the bug is needed.

Developers

Right now we have developer profiles from StackOverflow, GitHub and Bugzilla. Users are 'merged' under a common Developer profile using their emails.

To add new profiles, a schema must be created, and Developer must reference it. Check the Developer documentation for more information.
