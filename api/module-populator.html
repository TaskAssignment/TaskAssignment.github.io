Module: populator

Module to populate data from StackExchange and GitHub.

Requires

  • module:mongoose
  • module:request

Methods

(inner) gitHubPopulate(option, specificUrl, callback, etag, finalUrl)

Basic flow for GitHub requests. This handles the pagination and eventual errors, so every GitHub request should be made though this function.
Parameters:
Name Type Default Description
option string The option that is being populated now. It's used to check the request status later. It can be Issue, Comment, Developer, etc.
specificUrl string The url that will be appended to the base one.
callback function The function to call when the results are received. This should handle the creation or the updating of models in the database.
etag string GitHub API Etag to avoid repeated requests
finalUrl boolean false If set to true, specificUrl will be considered the endpoint of the request (without any other aditional parameters). If false, specificUrl will be added to the base url 'https://api.github.com/repositories' and will have a 'per_page' attribute added to the end.
See:
  • GitHub API overview

(inner) populateAnswers(ids, projectId, filter, obj, site)

Populate answers from StackOverflow using the soPopulate function. Answers can be populated based on users or not.
Parameters:
Name Type Default Description
ids string A string with up to 100 StakOverflow ids. This will be considered users ids if the obj is 'users/'.
projectId int Id of a GitHub project. This is used to filter the users on saving, since some ids are repeated.
filter string default The filter to be used on the StackExchange API.
obj string users/ If 'users/' will get the answers related to given user ids. If 'questions/' will get answers related to given questions. If '' will get answers with the given ids.
site string stackoverflow The StackExchange network to pull the data from.

(inner) populateComments(projectId, type)

Fetch comments from a repository. They may be on issue or commits, depending on what is specified by who calls this function. Commit Comments cannot be sorted, or filtered. So, when calling this for commits comments, it will always get ALL the comments on commits of this repository! Issue Comments can be filtered by date, so only new ones are fetched when calling this for issue comments.
Parameters:
Name Type Description
projectId int Id of a GitHub project.
type string Issue or Commit
See:
  • Commit Comments API
  • Issue Comments API

(inner) populateEvents(projectId)

Fetch all Issue Events. Issue Events are the 'history' of that issue and show when it was referenced by a commit (or comment), when it was closed, opened, assigned (or deassigned) to someone.

Name Type Description

(inner) populateIssues(projectId)

Populate Issues from a specific project on GitHub. It creates 'tags' based on the text of the issue and on the StackOverflow tags. On GitHub, Pull Requests are issues and they are fetched in this same function, they are saved in the database with the 'isPR' attribute equals to true. To avoid throttle problems, only the new issues are fetched (once there is at least one in the database).
Parameters:
Name Type Description
projectId int Id of a GitHub repository.
See:
  • Issues API

(inner) remakeRequest(options, requestCallback)

When some errors occur, the populate functions will reattemp to make the request. If the error persists, this will terminate the requests to avoid throttle problems.
Parameters:
Name Type Description
options object The options for the request. This may include the type of request, compression, url and other things.
requestCallback function The function to handle the requests. This is the responsible for pagination and sending the results to their specific handlers.
